Exfiltrating data from Android devices via WebView Takeover (Open Redirect)

Inês Martins - 06 Oct 2022

In this article, I go through the scenarios in which I've been able to exfiltrate data from real Android applications, after detecting a WebView takeover (aka "Open Redirect") vulnerability.

Typically this vulnerability is the result of allowing an external Intent to control a URI that is loaded onto a WebView, declared inside the vulnerable application.

This Intent can sometimes be triggered via deep link, but more often than not, it requires that a malicious application be installed in the victim's device.

Note that this article closely relates to the "Exploiting Deep Links in Android" series, particularly with Part 2.

1. Abusing Javascript Bindings

If the vulnerable WebView registers any web interfaces, and you're able to redirect the WebView to a website you control, then you can simply add JavaScript to your website that interacts with the app's code directly.

If you can read the app's source code, search for any calls to the  addJavascriptInterface method, to find the name of the interface and then you can find all of the exposed methods annotated with @JavascriptInterface.

On your website's JS code, you can call all of the exposed methods by using the window.InterfaceName.methodName syntax.

The impact will vary depending on what the exposed methods allow you to do.

To see a more detailed example, check out this HackerOne report: https://hackerone.com/reports/401793.

2. Reading local malicious file (< 24 SDK)

If the vulnerable WebView accepts file:// URIs, and the application can be run in devices with SDK < 24, you might be able exfiltrate all of the app's files with a little help from a malicious application.

Here are the steps:

  1. the malicious application installed in the victim's device writes an HTML file to a shared location accessible by the vulnerable app;
  2. the malicious application sends an Intent to the vulnerable app that contais the file:// URI that matches the location of the HTML file;
  3. the file:// URI is loaded by the vulnerable app;
  4. once the HTML file is loaded on the WebView, the JS code declared inside reads all local files and sends them to a server you control.

Note that the following conditions need to be present in order for this to work:

  • the malicious app needs to be granted the READ_EXTERNAL_STORAGE permission;
  • the vulnerable app needs to be granted the READ_EXTERNAL_STORAGE permission;
  • the vulnerable WebView needs to have JS enabled;
